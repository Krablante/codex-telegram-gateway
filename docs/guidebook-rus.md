# Spike + Omni: guidebook для новичка

Это короткий практичный гид по боту. Не каталог всех флагов, а нормальная стартовая карта: где работать, когда жать меню, что вообще не надо держать в голове, и какие команды реально стоит помнить.

Главная идея простая: для повседневной работы лучше опираться на меню, а не на память. Команды остаются полным интерфейсом, но меню и кнопки уже закрывают большую часть частых настроек.

## Быстрый старт

`General` - это лобби. Там обычно делают три вещи: открывают глобальное меню, запускают из него общие операторские действия и создают новые рабочие темы.

Обычный старт выглядит так:

```text
/new Backend Cleanup
```

Потом заходишь в новую тему и пишешь задачу обычным текстом. Это и есть нормальный happy path. Не работай в `General` как в обычной сессии и не сваливай несколько несвязанных задач в одну тему.

## Кто тут кто

`Spike` - основной worker. Он читает код, правит файлы, запускает команды и отвечает по задаче.

`Omni` нужен только для `/auto`. Это supervisor, а не второй тяжёлый worker. Он следит за циклами `Spike` и решает, продолжать ли задачу дальше.

Если ты только начинаешь, почти всегда достаточно одного `Spike`. `/auto` полезен позже, когда нужна длинная автономная задача и ты понимаешь, зачем за неё платить токенами.

## Почему лучше начинать с меню

Бот давно уже не про «помни двадцать команд наизусть». Базовый путь такой:

- в `General` пользуйся `/global` для глобальных настроек;
- в рабочей теме пользуйся `/menu` для настроек именно этой темы;
- в `Zoo` почти всё делай через pinned menu и inline-кнопки.

Через меню удобнее и безопаснее делать то, что меняется регулярно, но не является самой задачей:

- смотреть `Status` внутри темы;
- менять язык интерфейса через global menu в `General`;
- включать и выключать `wait`;
- открывать `Bot Settings` и там менять `Spike`/`Omni` model и reasoning, а также model/reasoning для `/compact`;
- запускать topic-операции через кнопки `Compact`, `Interrupt`, `Purge`;
- включать и выключать suffix;
- открывать `Zoo` или запускать `Clear` прямо из глобального меню в `General`;
- открывать `Guide` и `Help` прямо из глобального меню в `General`;
- обновлять экран кнопкой `Refresh`.

Практическая рекомендация простая: если тебе надо поменять стабильную настройку, сначала пробуй меню. Это быстрее, чище и обычно не требует помнить синтаксис. Отдельно полезно помнить несколько layout-деталей: глобальное меню теперь начинается с `Bot Settings` и `Language` в верхней строке, сразу под ними стоят `Guide` и `Help`, ниже остаются строки `Wait` / `Suffix` и `Zoo` / `Clear`, а bot-настройки по-прежнему вынесены в отдельный `Bot Settings` screen вместо прямой россыпи по root-экрану. Сами effective bot summary тоже остаются компактными, в духе `spike: gpt-5.4 (xhigh)`.

## Чего в меню нет

Меню не заменяет саму работу. Есть вещи, которые по смыслу остаются текстом или отдельными командами:

- обычный prompt в рабочей теме;
- `/new`, потому что тема создаётся командой;
- `/guide`, `/zoo` и `/clear` по-прежнему можно запускать из `General` прямыми командами;
- `Zoo`, `Clear`, `Guide` и `Help` доступны кнопками в глобальном меню, но прямые команды всё так же работают;
- `/q`, когда надо явно поставить следующий prompt после текущего;
- `/diff`;
- `/auto` и вопросы к `Omni`.

Ещё один важный момент: некоторые значения запускаются из меню, но вводятся не кнопкой. Например, custom wait или произвольный suffix text обычно приходят reply'ем на menu-сообщение после того, как бот попросил ввод.

Коротко: меню отлично подходит для переключателей, пресетов, экранов и типовых shortcut-действий в `General`. Свободный текст и реальные задачи остаются вне него.

## Как обычно идёт работа в теме

В рабочей теме обычный текст считается обычным prompt'ом. Если run свободен, он стартует сразу. Если run ещё жив, follow-up может пойти в тот же поток как продолжение. Если у live steer случается короткий временный сбой, бот сначала немного ретраит его и только потом сохраняет такой follow-up как следующий prompt вместо тупого generic busy-reply.

`/q` нужен не для «докинуть мысль», а для явного «сделай это следующим».

```text
/q После текущей проверки подготовь patch для README.
```

Если ничего не запущено, такой prompt обычно стартует сразу. Если текущий run ещё закрывается, `/q` спокойно ждёт своей очереди. Во время `/auto` эта очередь не работает, потому что маршрутизацией уже управляет `Omni`.

## Когда нужен /wait

`/wait` нужен, когда ты хочешь собрать один большой prompt из нескольких сообщений. Например: сначала отправить контекст, потом кусок лога, потом ещё уточнение, а уже потом всё одним пакетом отдать в `Spike`.

```text
/wait 60
```

После этого отправляешь несколько сообщений и потом отдельным сообщением делаешь flush. Подойдут:

```text
Все
Всё
All
```

Для повседневной работы чаще хватает локального `/wait`. Если надо одно и то же окно на весь чат, есть `/wait global`.

## Зачем нужен /suffix

`/suffix` - это не разовая подсказка, а постоянная привычка ответа. Его стоит использовать для правил вроде:

- отвечай кратко;
- всегда говори, что проверил;
- не переводи команды и код;
- перед финалом отмечай, запускал ли тесты.

Если правило нужно только в одной теме, используй topic suffix. Если это глобальная привычка для всего чата, используй global suffix. Для одноразовой инструкции suffix обычно не нужен: проще написать это прямо в prompt.

## Zoo без лишнего веса

`/zoo` - это не рабочая тема для `Spike`, а отдельная операторская панель по проектам. Она нужна, чтобы быстро понять, какой репозиторий выглядит здоровым, какой просел и куда логичнее копать дальше.

Нормальный сценарий в Zoo почти menu-only:

1. Открываешь `/zoo`.
2. Жмёшь `Add project`.
3. Описываешь проект обычными словами.
4. Бот предлагает найденный root.
5. Отвечаешь `Yes` или `No`.

После этого в теме живёт одна pinned-панель с карточками питомцев и кнопками вроде `Back`, `Refresh`, `Remove`, `Respawn menu`. Сами кнопки в Zoo намеренно остаются на английском даже в русском интерфейсе.

Что важно помнить:

- Zoo не для обычных prompt'ов и не для coding-сессий;
- `Refresh` обновляет карточку проекта без лишнего спама в тему;
- `Remove` убирает питомца из Zoo, но не трогает репозиторий;
- если проектов много, список листается страницами.

Этого достаточно. Не надо думать о Zoo как о втором командном интерфейсе: это скорее компактный status board по репам.

## Когда реально нужен /auto

`/auto` нужен для длинной автономной задачи, а не для обычного диалога. Ты задаёшь цель, даёшь стартовый prompt, дальше `Omni` следит за завершёнными циклами `Spike` и решает, что делать дальше.

Если тебе просто нужно обычное взаимодействие с ботом, `/auto` почти точно не нужен. Сначала освой обычную тему со `Spike`, `/menu`, `/diff` и `/compact`, а потом уже подключай `Omni`, если от него есть практический выигрыш.

## Что реально стоит помнить

Если не хочется держать в голове много команд, на старте достаточно этих:

- `/new` - создать рабочую тему;
- `/global` - открыть глобальное меню в `General`;
- `/menu` - открыть локальное меню темы;
- `/status` - быстро посмотреть состояние темы;
- `/q` - поставить следующий prompt в очередь;
- `/diff` - посмотреть, что уже изменилось;
- `/compact` - сжать длинную историю в brief;
- `/guide` - получить PDF-гайдбук;
- `/zoo` - открыть проектную панель;
- `/interrupt` - остановить run, если это правда нужно.

Остальное можно спокойно добирать по мере необходимости через help и меню.

## Хорошие привычки

Работай по схеме «одна тема = один поток работы». Не используй `General` как обычную рабочую сессию. Для настроек сначала смотри в меню. Для текста и самой задачи используй обычные сообщения. Для явного «сделай это потом» используй `/q`. Если тема разрастается, не тяни с `/compact`.

И ещё одна практическая мелочь: если оператор в этот момент перекатывает сервис, нормальный активный run обычно должен спокойно добежать до конца, а не оборваться посередине.

Этого уже достаточно, чтобы нормально пользоваться ботом без погружения во внутренности.
