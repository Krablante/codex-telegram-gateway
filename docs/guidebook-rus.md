# Spike + Omni: guidebook для новичка

Это короткий человеческий гид по боту. Не reference по каждому флагу, а нормальная карта местности: кто тут вообще работает, где писать, когда использовать `/q`, `/wait`, `/suffix`, что делает `/auto`, и в каких случаях нужны `/diff`, `/compact` и `/purge`.

## С чего начать

Думай о системе просто: `General` - это лобби, а рабочие темы - это отдельные комнаты для задач. В `General` ты открываешь глобальные настройки, просишь справку и забираешь guidebook. Реальная работа почти всегда живёт не там, а в конкретной теме.

Самый нормальный старт выглядит так: создай тему сразу командой вида `/new Topic Name`, зайди в неё и напиши обычный prompt обычным текстом. Если задача длинная, время от времени смотри `/diff` и делай `/compact`.

Пример:

```text
/new Backend Cleanup
```

Самая полезная мысль для новичка: одна тема - один поток работы. Не сваливай туда пять несвязанных задач сразу.

## Кто тут кто

`Spike` - это основной рабочий. Он получает обычные prompt'ы, читает код, правит файлы, запускает команды и присылает финальный ответ. Если ты просто пришёл в тему и начал писать задачу, почти наверняка ты работаешь именно со `Spike`.

`Omni` нужен только для `/auto`. Он не подменяет `Spike` и не делает тяжёлую работу вместо него. Его роль - держать цель, смотреть на завершённые циклы `Spike` и решать, что делать дальше: продолжать, подождать, сменить proof line, заблокироваться или закончить.

Для старта всей системы `Omni` вообще не обязателен. Если у тебя лимитный тариф вроде `ChatGPT Plus`, часто разумно начать только со `Spike`: автоматический режим через `Omni` обычно тратит заметно больше токенов и быстрее сжигает лимиты.

Если совсем коротко: `Spike` - live worker, а `Omni` - supervisor для `/auto`.

## Где писать и что произойдёт

В рабочей теме обычный текст считается обычным prompt'ом. Пока run свободен, он стартует сразу. Если run ещё жив, follow-up может попасть в тот же рабочий поток как продолжение. Это не ограничено одной дополнительной репликой: пока live run ещё открыт, можно продолжать докидывать follow-up'ы, и они будут steer'иться в тот же самый run.

`/q` нужен для другой ситуации: когда ты не хочешь просто "докинуть мысль", а хочешь явно сказать боту "сделай это следующим после текущего". Тогда prompt идёт в очередь `Spike`.

Пример:

```text
/q После текущей проверки подготовь patch для README.
```

Если в теме сейчас ничего не запущено, такой prompt обычно стартует сразу. Если предыдущий run ещё только дописывает финал и закрывается, `/q` спокойно дождётся своей очереди и запустится следующим. Во время `/auto` эта механика отключена: там маршрутизацией уже управляет `Omni`.

## Когда нужен /wait

`/wait` полезен не для очереди, а для сборки одного большого prompt из нескольких сообщений. Это удобно, если ты хочешь сначала скинуть контекст кусками, потом приложить файл, а потом всё отправить в `Spike` как один аккуратный запрос.

Пример:

```text
/wait 60
```

После этого можно спокойно отправить несколько сообщений подряд, а в конце отправить отдельное flush-сообщение. Подойдут любые из этих слов:

```text
Все
Всё
All
```

`/wait global` делает то же самое на уровне всего чата, но для повседневной работы чаще всего хватает локального одноразового окна.

## Зачем вообще нужен /suffix

`/suffix` - это постоянная инструкция, которая автоматически дописывается к prompt'ам. Он нужен не для одноразовых пожеланий, а для привычек и стабильных правил ответа. Хороший suffix обычно звучит так: "отвечай кратко", "всегда говори, что проверил", "оставляй команды и код без перевода", "перед финалом сообщай, запускал ли тесты".

Примеры:

```text
/suffix Отвечай кратко, по делу и всегда говори, какие тесты ты запускал.
```

```text
/suffix global Keep answers concise. Always call out verification.
```

Если правило нужно только в одной теме, используй topic suffix. Если это общая привычка для всего чата, используй global suffix. Если инструкция нужна один раз, suffix не нужен - проще написать обычный prompt.

## Полезные команды без перегруза

Многие команды вообще не обязательно помнить и вводить руками. В `General` есть global menu с общими настройками, а внутри рабочей темы есть local menu с настройками именно этой темы. Через эти меню удобно менять язык, wait-окна, модели и другие частые параметры без ручного набора команд.

- `/help` показывает список команд и короткие пояснения. Если какая-то команда непонятна или ты просто хочешь увидеть весь список, начни именно с `/help`.
- `/limits` показывает текущее состояние лимитов Codex: live окна `5h` и `7d` или явный безлимитный режим.
- `/status` показывает, в каком состоянии тема и какие model/reasoning реально применяются сейчас.
- `/menu` открывает локальное меню настроек темы, включая встроенный экран `Status`. При повторном открытии старое меню аккуратно заменяется, а телеграмный вариант `/menu@YourBot` тоже поддерживается.
- `/language` меняет язык интерфейса.
- `/model` и `/reasoning` управляют самим `Spike`.
- `/omni_model` и `/omni_reasoning` управляют `Omni`, если `/auto` вообще используется в этом деплойменте.

Для длинной задачи чаще всего достаточно помнить ещё три команды:

- `/diff` быстро показывает, что уже изменилось в workspace.
- `/compact` пересобирает длинную историю темы в короткий рабочий конспект. Бот сохраняет суть того, что уже сделано и что важно дальше, чтобы можно было продолжать работу без огромной простыни старых сообщений.
- `/purge` сбрасывает локальную память темы, если она больше не нужна и ты хочешь начать почти с нуля.

## Как думать про /auto

`/auto` нужен для длинной автономной задачи, а не для обычного диалога. Ты включаешь `/auto`, задаёшь цель, даёшь стартовый prompt, а дальше `Omni` следит за завершёнными циклами `Spike` и аккуратно двигает задачу вперёд. Помни три вещи: главный worker всё ещё `Spike`, операторский диалог в теме идёт через `Omni`, а `/q` в этом режиме не нужен и не работает как обычная очередь.

Если тебе просто нужно спросить "что уже сделано?" или "почему сейчас пауза?", это можно делать прямо в той же теме.

## Хорошие привычки

Не пытайся держать всё в одной теме. Не используй `General` как рабочую сессию. Если ты точно хочешь "сделай это следующим после текущего" - используй `/q`, а не надежду на тайминг. Если задача распухает, делай `/compact` раньше, чем тема превратится в свалку. Если нужна стабильная манера ответа, выноси её в `/suffix`, а не повторяй в каждом prompt.

Этого уже достаточно, чтобы уверенно начать пользоваться ботом без погружения во внутренности.
